1 System Overview & Design Goals

1.1 System Purpose

ORVIKA AI is a Windows desktop application that provides AI-powered document analysis, video/audio transcription, and hybrid research capabilities while guaranteeing that user documents never leave the local device. The system processes sensitive data locally using a quantized large language model (Gemma 4) and selectively queries cloud AI services for public research only.

1.2 Design Goals

GoalDescriptionPriority
Privacy GuaranteeZero document content transmitted externally. Architecturally enforced, not policy-enforced.Critical
TransparencyEvery outbound network request logged, visible, auditable by the user in real-time.Critical
Offline CapabilityAll core features (chat, document analysis, transcription) functional without internet.Critical
Hybrid IntelligenceSeamless combination of local reasoning and cloud research with clear source attribution.High
PerformanceSub-5-second first token latency (GPU), sub-10-second document parsing (50-page PDF).High
ExtensibilityModular architecture supporting new model runtimes, document types, and cloud providers.Medium

1 Architecture Overview

1.1 Architectural Style

The system follows a layered monolithic architecture with clear separation between presentation, application logic, processing, connectivity, and persistence layers. This is a single deployable unit (desktop application) — not a microservice architecture — because:

  • Single-user desktop application with no network service requirements
  • All processing happens locally; no horizontal scaling needed
  • Minimizes deployment complexity for end users
  • Enables tight integration between components for performance

Within the monolith, components communicate via Rust trait interfaces (compile-time polymorphism) and Tauri IPC commands (frontend ↔ backend).

3.1 sessions

CREATE TABLE sessions (
    id              TEXT PRIMARY KEY,         -- UUID v4
    name            TEXT NOT NULL,            -- User-provided or auto-generated
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at      TEXT NOT NULL DEFAULT (datetime('now')),
    research_mode_enabled BOOLEAN NOT NULL DEFAULT 0,
    cloud_provider  TEXT,                     -- 'openai' | 'gemini' | 'claude' | NULL
    privacy_level   TEXT NOT NULL DEFAULT 'balanced'
                    CHECK (privacy_level IN ('strict', 'balanced', 'permissive')),
    model_id        TEXT NOT NULL,            -- References model_downloads.id
    is_active       BOOLEAN NOT NULL DEFAULT 1,
    metadata        TEXT                      -- JSON: custom tags, notes

CREATE INDEX idx_sessions_updated ON sessions(updated_at DESC);
CREATE INDEX idx_sessions_active ON sessions(is_active) WHERE is_active = 1;

    id              TEXT PRIMARY KEY,         -- UUID v4

    content         TEXT NOT NULL,
    source_type     TEXT CHECK (source_type IN ('local', 'research', 'mixed', NULL)),
    sources         TEXT,                     -- JSON array of SourceCitation objects
    created_at      TEXT NOT NULL DEFAULT (datetime('now')),
    tokens_used     INTEGER,
    latency_ms      INTEGER,
    metadata        TEXT                      -- JSON: model used, temperature, etc.
);

CREATE INDEX idx_messages_session ON messages(session_id, created_at);
CREATE INDEX idx_messages_role ON messages(role);

SQLite schema migrations are managed via a sequential numbered migration system stored in src-tauri/migrations/. Each migration is a pair of SQL files:

  • 001_create_sessions.sql — Forward migration
  • 001_create_sessions.down.sql — Rollback migration

Migrations run automatically on application startup. A schema_migrations table tracks applied versions:

CREATE TABLE schema_migrations (
    version     INTEGER PRIMARY KEY,
    name        TEXT NOT NULL,
    applied_at  TEXT NOT NULL DEFAULT (datetime('now')),
    checksum    TEXT NOT NULL             -- SHA-256 of migration file
);

6.2 Vector ID Mapping

FAISS assigns sequential integer IDs to vectors. The mapping between FAISS vector IDs and document chunk IDs is maintained in the embedding_id column of the document_chunks table. When a chunk is added to FAISS, the returned vector ID is stored in the chunk record.
